Proposta de Solução

⏱ 3 minutos 🎤 Integrante 3
🎤 Roteiro de Fala

"Nossa proposta é um sistema digital de controle de acesso baseado em QR Code — funcionando 100% pelo celular, sem necessidade de nenhum hardware especial além de um smartphone."

"O fluxo começa pelo próprio participante: ele abre o app, digita o e-mail e se cadastra sozinho. Depois, escolhe o evento desejado e faz uma solicitação de vaga. O administrador recebe essa solicitação no painel e decide aprovar ou rejeitar. Se aprovado, o sistema gera automaticamente um ingresso digital exclusivo — um QR Code único baseado em UUID."

"Na hora do evento, o controlador de acesso abre o app na tela de scanner, aponta a câmera para o QR Code do participante e, em menos de um segundo, recebe a resposta: entrada liberada ou negada. O sistema também registra a saída — assim o organizador sabe exatamente quem está dentro do evento em cada momento."

"O sistema é inteligente: bloqueia tentativas de dupla entrada, controla a capacidade máxima do evento e impede reentrada após a saída. Um participante que saiu não pode mais entrar novamente com o mesmo ingresso."

Banco de Dados

⏱ 2 minutos 🎤 Integrante 5
🎤 Roteiro de Fala

"O banco de dados do nosso sistema é composto por 5 tabelas relacionais no PostgreSQL, cada uma com uma responsabilidade bem definida."

"A tabela central é a de ingressos — ela é o coração do sistema. Cada ingresso possui um token UUID gerado aleatoriamente pelo servidor no momento da criação. Esse token é o que vira o QR Code do participante. Ele é único, imprevisível e nunca se repete."

"A tabela de leituras registra cada tentativa de scan — válida ou inválida, de entrada ou saída. Isso nos dá um histórico completo de auditoria: quem entrou, quem saiu, quando e por qual dispositivo."

"Uma regra importante: o status do ingresso segue um ciclo de vida bem definido — começa como PENDENTE, muda para DENTRO quando o participante entra, e finaliza como SAIU quando ele sai. Uma vez em SAIU, o ingresso está encerrado e não permite reentrada."

"A tabela de solicitações representa o pedido de vaga feito pelo próprio participante — ela guarda o estado PENDENTE, APROVADO ou REJEITADO de cada pedido antes que o ingresso seja gerado."

Tabela Descrição Campos Principais
📅 evento Eventos criados pelos organizadores id, nome, local, capacidade, data
👥 participante Pessoas cadastradas — podem se auto-registrar pelo app id, nome, email
📝 solicitacao Pedido de vaga de participante para evento id, status (PENDENTE/APROVADO/REJEITADO), participante_id, evento_id, criada_em
🎫 ingresso Ingresso gerado após aprovação — contém o QR Code id, token (UUID único), status (PENDENTE/DENTRO/SAIU), evento_id, participante_id
📋 leitura Histórico de todas as tentativas de scan id, resultado (VALIDO/INVALIDO/JA_USADO/LOTADO/NAO_ENTROU), tipo (ENTRADA/SAIDA), dispositivo, lido_em
✅
Ponto de destaque: O token UUID é gerado exclusivamente pelo servidor no momento da aprovação da solicitação — nunca pelo app. Isso significa que não existe forma de um usuário criar ou falsificar um token válido. O ciclo completo é: solicitação → aprovação admin → ingresso com UUID → QR Code → entrada → saída.

Backend — API Spring Boot

⏱ 3 minutos 🎤 Integrante 6
🎤 Roteiro de Fala

"O backend é uma API REST desenvolvida com Spring Boot 3.2, Java 17. Ele roda num servidor local e expõe todos os endpoints que o app Android consome via HTTP/JSON usando a biblioteca Retrofit."

"A parte mais importante é o serviço de check-in. Quando o scanner lê um QR Code, o app envia o token e o tipo de operação — ENTRADA ou SAÍDA — para o endpoint /api/checkin. O backend executa uma sequência de verificações: o token é válido? O ingresso está no estado correto para essa operação? O evento ainda tem capacidade? Dependendo das respostas, retorna VÁLIDO, JÁ_USADO, LOTADO, NAO_ENTROU ou INVÁLIDO. Toda tentativa é registrada na tabela de leituras."

"Outro serviço importante é o de solicitações: quando um participante pede uma vaga, a solicitação é salva com status PENDENTE. O admin aprova ou rejeita. Na aprovação, o sistema gera automaticamente o ingresso com um token UUID aleatório e único."

"O backend segue o padrão em camadas: Controller → Service → Repository. Os controllers recebem as requisições HTTP, os services contêm a lógica de negócio, e os repositories fazem a comunicação com o banco via JPA/Hibernate."

Método Endpoint Descrição
POST/api/eventosCriar novo evento
GET/api/eventosListar todos os eventos
PUT / DELETE/api/eventos/{id}Editar / Excluir evento (com cascade)
POST/api/participantesCadastrar participante (auto-registro)
GET/api/participantes/email/{email}Buscar participante por e-mail (login)
POST/api/solicitacoesParticipante solicita vaga em evento
GET/api/solicitacoes/pendentesAdmin lista solicitações pendentes
PUT/api/solicitacoes/{id}/aprovar⭐ Admin aprova → gera ingresso UUID
PUT/api/solicitacoes/{id}/rejeitarAdmin rejeita solicitação
POST/api/checkin⭐ Validar QR Code (ENTRADA ou SAIDA)
GET/api/leituras/evento/{id}Histórico de leituras (dashboard)
GET/api/relatorio/evento/{id}⭐ Relatório de presença ao vivo
💡
Dica de fala: Se a banca perguntar sobre segurança, mencione que o token UUID v4 tem 2¹²² combinações possíveis — é estatisticamente impossível adivinhar um token válido por força bruta. E toda tentativa — válida ou inválida — fica registrada permanentemente no banco com dispositivo e horário.

Desafios e Aprendizados

⏱ 3 minutos 🎤 Integrante 10
🎤 Roteiro de Fala

"Todo projeto de desenvolvimento enfrenta desafios, e com o nosso não foi diferente. Quero compartilhar os principais obstáculos que superamos, porque acredito que eles mostram o aprendizado técnico real do grupo."

🔴 Desafios Encontrados

1
Incompatibilidade do Lombok com Java 17+ O backend não compilava — dezenas de erros de "método não encontrado". A biblioteca Lombok, que gera getters/setters automaticamente via anotações, se mostrou instável com a versão do Java usada no projeto.
2
Constraints legadas bloqueando operações no banco O PostgreSQL rejeitava inserções com erro de constraint — o banco tinha restrições antigas de valores válidos que conflitavam com os novos status do sistema (DENTRO, SAIU, NAO_ENTROU).
3
IP dinâmico quebrando a conexão a cada rede Cada vez que o notebook mudava de rede Wi-Fi, o IP mudava e o app Android parava de se comunicar com o backend — exigindo recompilação do app para atualizar a URL.
4
Exclusão de eventos causando erro de chave estrangeira Ao tentar excluir um evento, o banco retornava erro de violação de FK — havia ingressos, leituras e solicitações vinculados ao evento que precisavam ser removidos primeiro, na ordem correta.
5
Race condition no scanner de QR Code O scanner processava o mesmo QR Code múltiplas vezes simultaneamente — threads diferentes liam o código antes que a requisição anterior terminasse, gerando validações duplicadas.

🟢 Como Resolvemos

1
Reescrita manual sem Lombok Substituímos todas as anotações Lombok por código Java explícito — getters, setters e construtores escritos manualmente. O projeto ficou mais verboso, mas 100% estável e sem dependências problemáticas.
2
SQL de inicialização automática (data.sql) Criamos um arquivo data.sql executado automaticamente pelo Spring Boot na inicialização, que remove as constraints legadas com ALTER TABLE DROP CONSTRAINT IF EXISTS. Sem necessidade de acesso manual ao banco.
3
Configuração dinâmica de IP via SharedPreferences Refatoramos o cliente HTTP para ler o endereço do servidor de uma preferência salva no app. O usuário toca num botão na tela inicial, digita o novo IP e reconecta — sem recompilar.
4
Deleção em cascata com @Transactional Implementamos no Service a exclusão em ordem correta: leituras → ingressos → solicitações → evento, tudo dentro de uma transação. Se qualquer passo falhar, o banco é revertido automaticamente.
5
AtomicBoolean para controle de concorrência Substituímos o boolean simples por AtomicBoolean com operação compareAndSet — garante que apenas uma thread processa o scan por vez, eliminando as leituras duplicadas.
🎤 Fechamento

"Cada um desses desafios nos ensinou algo que vai muito além da teoria: depurar erros de banco de dados, lidar com concorrência em ambientes mobile, entender segurança de rede no Android. Esses são os problemas que todo desenvolvedor encontra no mercado — e resolvê-los aqui nos preparou para isso."

💬 Possíveis Perguntas da Banca — Respostas Preparadas

?
"Como garantem que o QR Code não pode ser falsificado?" O token é um UUID v4 gerado aleatoriamente pelo servidor somente após a aprovação da solicitação, com 2¹²² combinações possíveis. Ele não é previsível, não existe antes da aprovação e só pode ser criado por quem tem acesso ao banco.
?
"Um participante pode entrar mais de uma vez ou reentrar após sair?" Não. O sistema tem três estados para o ingresso: PENDENTE (aguardando entrada), DENTRO (dentro do evento) e SAIU (encerrado). Tentativa de segunda entrada com status DENTRO ou SAIU é bloqueada com mensagem específica e registrada no histórico.
?
"O que acontece se o sistema cair durante o evento?" Esta é uma limitação do MVP atual. Como melhoria futura, planejamos implementar um cache local no app scanner para operar em modo offline por curtos períodos, sincronizando quando a conexão voltar.
?
"Como vários scanners usam o sistema ao mesmo tempo?" O controle de concorrência ocorre em duas camadas: no app, usamos AtomicBoolean para evitar processamento duplo no mesmo dispositivo. No servidor, as operações são transacionais no banco — dois scanners simultâneos não conseguem validar o mesmo ingresso duas vezes.
?
"Por que Java e não Kotlin para o Android?" Java é a linguagem base que estudamos na disciplina. O Android suporta ambas nativamente — a migração para Kotlin seria um próximo passo natural, pois o código Java é 100% interoperável com Kotlin.
?
